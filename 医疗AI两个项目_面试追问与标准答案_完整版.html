三、项目一高频追问与标准答案

Q1：你说用了 RAG，完整流程是什么？

用户输入问题后，系统先做意图识别，比如判断是科室推荐、药品说明、急救知识、费用查询还是药物相互作用检查。对于知识类问题，系统会进入 RAGFlow 知识库检索，从疾病症状库、药品说明书库或急救知识库中召回相关片段。

召回结果会和用户问题一起拼接进 prompt，让 LLM 基于检索到的资料生成回答。最后输出时会追加免责声明；如果识别到高危症状，比如胸痛、呼吸困难、意识模糊，会优先输出急诊或 120 提示。

Q2：为什么不用大模型直接回答？

医疗场景对准确性和可控性要求更高。直接让大模型自由回答，容易出现幻觉，比如编造医生信息、错误解释药品禁忌、或者给出过于确定的诊断结论。

RAG 的价值是先检索医院已有资料或可信知识库，再让大模型基于资料回答。这样回答范围更可控，也更容易追溯来源。我的理解是：LLM 负责语言组织和解释，知识库负责事实约束。

Q3：知识库里放了什么？

主要包括科室介绍、医生专长、常见疾病与科室关系、药品说明书、急救知识和一些挂号说明。比如用户问“牙痛挂什么科”，可以召回口腔科和相关疾病资料；用户问“布洛芬怎么吃”，可以召回药品说明书里的用法、禁忌、最大剂量等内容。

Q4：文档怎么切分？chunk 太大或太小有什么问题？

切分思路是按业务语义切，而不是机械按字数切。比如药品说明书可以按“适应症、用法用量、不良反应、禁忌、注意事项”切；科室介绍可以按“科室职责、常见疾病、医生专长、挂号建议”切。

chunk 太大，召回内容会包含很多无关信息，prompt 里噪声变多；chunk 太小，容易丢失上下文，比如只召回“禁用”但没有召回具体药品或人群。实际工程里我会先用中等大小切分，再通过典型问题测试召回效果。

Q5：如何处理症状模糊，比如“肚子疼”？

症状模糊时不能直接给确定科室。比如“肚子疼”可能涉及消化内科、普外科、妇科、泌尿外科甚至急诊。系统应该先追问关键补充信息，比如疼痛位置、持续时间、是否发热、呕吐、腹泻、是否怀孕、是否剧烈疼痛等。

我的回答策略是：先给可能方向，再提示需要补充信息。如果出现剧烈腹痛、持续加重、伴随休克表现等高危信号，优先建议急诊。

Q6：你说提升准确率，有没有数据？

这个问题不能硬编数字。更稳的回答是：这个项目我没有把它包装成严格的模型准确率评估，而是从问答相关性和业务可用性角度做验证。我们会准备典型挂号问题、药品问题、急救问题，检查系统是否召回正确资料、回答是否符合业务边界、是否追加免责声明和高危提示。

如果后续做正式评估，我会建立标准测试集，分别统计检索命中率、回答相关性、人工评估通过率、幻觉率和高危规则触发准确率。

Q7：Coze / Dify 里你到底做了什么，不会只是拖节点吧？

我做的不只是拖节点，而是围绕业务流设计节点之间的数据流转。比如先识别用户意图，再根据意图进入不同分支：药品问题走药品说明书知识库，科室推荐走疾病症状库和科室表，费用问题走 MySQL 服务价格表，高危症状走规则引擎。

此外，还要处理节点输出格式、变量传递、知识库召回参数、prompt 约束、兜底回复和前端展示字段。平台只是实现工具，核心是把医疗业务规则拆成可执行的流程。

Q8：医疗建议如何做风险控制？

第一，不输出确定诊断，只输出导诊建议和风险提示。第二，所有医疗建议统一追加免责声明，说明不能替代医生诊断。第三，高危症状优先触发急诊或 120 提示，例如胸痛、呼吸困难、意识模糊、严重外伤、中风样症状等。第四，信息不足时引导用户补充症状，而不是强行给出结论。

五、数据工程追问与标准答案

Q1：原始数据规模是多少？你怎么处理的？

原始问诊文本大约 25 万条，覆盖 58 类科室，存在明显长尾问题。处理方式不是直接训练，而是先做数据清洗：去空文本、无效字符、单字样本；然后处理同文本不同标签的冲突样本；再做去重，避免重复模板污染指标。

清洗后再做小样本科室补齐，针对低资源科室构造症状、疾病、部位、伴随表现等 profile，生成增强样本，并再次做去重和科室名防泄漏。

Q2：什么是标签冲突？为什么要处理？

标签冲突是指完全相同或高度相似的问诊文本，被标注成不同科室。比如同一句“肚子疼、恶心、反酸”，一条标消化内科，另一条标普通外科。

如果不处理，模型会收到互相矛盾的监督信号，训练时不知道该往哪个方向收敛。处理方式是先识别冲突样本，有人工条件时人工复核；没有人工条件时，优先删除高冲突样本，保证监督信号更干净。

Q3：为什么要防止科室名泄漏？

如果训练文本中直接出现“骨科”“眼科”“皮肤科”这样的科室名，模型会学到捷径，而不是学习症状和科室之间的真实关系。测试指标可能很好看，但真实用户一般不会说“我要挂骨科”，而是说“膝盖疼”“眼睛红”“皮肤起疹子”。

所以我会清理文本中直接泄漏标签的科室名，或者把这类样本单独处理，避免模型靠关键词作弊。

Q4：为什么长尾问题严重？

医疗科室分布天然不均衡。儿科、皮肤科、妇产科这类高频科室样本很多，而一些专科或细分科室样本很少。模型训练时容易偏向样本多的头部类，导致尾部类召回率低。

所以这个项目没有只看 Accuracy，而是重点看 Macro F1。Accuracy 容易被头部类主导，Macro F1 是每个类别分别计算后再平均，更能反映尾部类别的表现。

Q5：你怎么做数据增强？如何避免语义漂移？

增强不是简单同义词替换，而是围绕科室 profile 生成更多问诊表达。比如针对某个小科室，整理常见症状、疾病、身体部位、伴随表现，再生成多样化口语问诊文本。

避免语义漂移主要靠三点：第一，增强样本必须围绕该科室真实疾病和症状；第二，生成后要去重、去空文本、去科室名泄漏；第三，增强数据不能替代真实测试集，最终要在独立测试集上验证。

Q6：训练集、验证集、测试集怎么切？如何避免数据泄漏？

通常按照 train/dev/test 切分，训练集用于模型学习，dev 集用于调参和保存最佳模型，test 集只在最终评估时使用。

防止数据泄漏要注意：重复样本不能同时出现在训练集和测试集；增强样本不能和测试集模板高度相似；同一来源、同一模板批量生成的文本最好在切分前先去重或按组切分。否则测试指标会虚高。

Q7：短文本如“头晕”“胸痛”是不是噪声？

不能简单当噪声。医疗场景下短文本可能非常重要，比如“胸痛”“咳血”“呼吸困难”。这类文本虽然短，但风险高。我的处理策略是：空文本、无意义字符、单字乱码可以清理；但有效短症状要保留，必要时进入高危规则或追问流程。

六、RoBERTa 追问与标准答案

Q1：为什么选 RoBERTa-wwm-ext？

RoBERTa-wwm-ext 比较适合中文短文本语义理解。wwm 是 whole word masking，中文里很多医学症状是词级概念，比如“呼吸困难”“胸口疼”“关节疼痛”，整词 Mask 能让模型更关注词级语义，而不是单个字。

相比原始 BERT，RoBERTa 训练更充分，取消 NSP 后更聚焦语言建模。在这个项目中，我用它承担头部和腰部科室的快速分类，结构是预训练 RoBERTa 主干加 Linear 分类头。

Q2：模型输入输出是什么？

输入是用户症状文本。经过 tokenizer 后变成 input_ids 和 attention_mask，统一 padding 或 truncation 到最大长度 194。模型输出是每个科室类别的 logits，然后对 logits 做 softmax，取最大概率对应的类别作为预测科室，同时把最大 softmax 概率作为工程上的置信度信号。

Q3：分类头怎么设计？用 CLS 还是 mean pooling？

这条线采用的是比较标准的 BERT/RoBERTa 分类结构：取 pooled output 或 CLS 表示，再接一个 Linear(768, num_classes) 分类头输出各科室 logits。优点是实现简单、推理快、适合固定类别分类。

如果继续优化，可以对比 CLS、mean pooling、attention pooling 的效果，但项目里优先采用稳定的标准分类头。

Q4：训练参数是什么？

RoBERTa 教师模型训练的核心参数是：batch size 82，最大长度 194，epochs 4，learning rate 5e-5，hidden size 768。优化器使用 AdamW，损失函数使用 CrossEntropyLoss。训练中每 500 个 batch 或 epoch 末尾在 dev 集评估，如果 dev Macro F1 更高，就保存最佳模型。

Q5：模型效果怎么样？

RoBERTa 这条线做的是 24 类科室分类，测试集大约 49,463 条。测试 Accuracy 约 0.8788，Macro Precision 约 0.8647，Macro Recall 约 0.8444，Macro F1 约 0.8497。

这个结果说明模型对头部和部分腰部科室比较稳定，但一些边界模糊或者样本较少的科室仍需要后续 RexUniNLU 或 LLM 复核。

Q6：为什么不用 LLM 直接分类？

LLM 直接分类成本更高、延迟更高、输出稳定性也不如固定分类器。高频分诊场景更适合用小模型快速处理，结果也更容易评估和部署。

LLM 更适合放在兜底层，用于信息不足、跨科室、需要多轮追问或生成解释的场景。

Q7：类别不均衡怎么处理？

处理类别不均衡可以从数据和训练两侧入手。数据侧做小样本科室补齐、去重、防泄漏；训练侧可以考虑 weighted loss、过采样、Focal Loss 等。项目里核心思路是先清洗和补齐，再用 Macro F1 评价模型是否照顾到尾部类别。

七、RexUniNLU 追问与标准答案

Q1：RexUniNLU 是什么？为什么引入它？

RoBERTa 是固定分类头，适合固定类别、高频科室快速预测。但医疗科室长尾严重，低资源科室样本少，RoBERTa 容易偏向头部类。所以我引入 RexUniNLU 作为第二层复核模型。

RexUniNLU 的特点是 schema-driven，也就是 schema 可以作为任务定义。项目里我把 20 个低资源科室写进 schema，让模型根据输入症状判断 [CLASSIFY] 这个位置属于哪个科室。它更适合做候选类别可配置、尾部类别复核的任务。

Q2：schema-driven 具体是什么意思？

schema-driven 指的是模型不是只依赖固定分类头，而是通过 schema 告诉模型当前要识别哪些类型。比如 schema 中包含“骨科、眼科、乳腺科、耳鼻喉科”等 20 个候选科室，模型就围绕这个 schema 做判断。

在项目里，我把分类任务改写成抽取任务：文本前面加 [CLASSIFY]，然后让模型判断 [CLASSIFY] 这个 span 的 type 是哪个科室。

Q3：RexUniNLU 的数据格式是什么？

原始分类数据是“文本 + 标签 ID”，但 RexUniNLU 需要 JSONL。每条数据包括 id、text、info_list 和 schema。text 前面加 [CLASSIFY]；info_list 里写真实科室标签；schema 里写完整候选科室列表。

{ "id": "train_0", "text": "[CLASSIFY]膝盖疼，走路疼，要挂什么科？", "info_list": [{ "type": "骨科", "span": "[CLASSIFY]", "offset": [0, 10] }], "schema": { "骨科": null, "眼科": null, "耳鼻喉科": null } }
Q4：RexUniNLU 训练参数和效果是什么？

RexUniNLU 微调的是 20 类低资源科室，训练 3 个 epoch，学习率 3e-5，batch size 4，warmup_ratio 0.1，dropout 0.5，max_len 194，hint_max_len 128。dropout 设置偏高，是为了缓解低资源类别过拟合。

测试集是 2400 条，20 类，每类约 120 条，Acc、Macro Precision、Macro Recall、Macro F1 都约 0.9971。这个结果很高，但面试时要强调：这是筛选出的 20 个语义锚点比较明确的低资源子任务，不等同于全 58 类真实场景。

Q5：RexUniNLU 为什么阈值设置得比 RoBERTa 更高？

RexUniNLU 在系统里不是常规快速通道，而是高门槛复核通道。它只在非常确定时接管输出，所以阈值设置为 0.95~1。医疗场景里误分诊成本高，宁愿多走一次 LLM 兜底，也不要让尾部复核模型在不够确定时强行输出。

Q6：如果 RexUniNLU 指标这么高，为什么还需要 LLM 兜底？

因为 RexUniNLU 的 0.9971 是在 20 类子任务测试集上的结果，不等于真实线上所有输入。真实用户可能输入跨科室症状、极短文本、无关问题、口语化表达、甚至高危急症。模型分类再高，也不能覆盖所有情况。

LLM 兜底不是为了追求更高分类分数，而是处理信息不足、多轮追问和风险提示。

八、知识蒸馏追问与标准答案

Q1：为什么要做知识蒸馏？

12 层 RoBERTa 教师模型效果更好，但线上部署时模型体积和推理延迟更高。医疗问诊希望响应更快，所以我设计 6 层学生模型，让学生保留一半 Transformer 层，通过蒸馏学习教师模型的输出分布和中间表示，用少量精度损失换取推理加速和部署成本下降。

Q2：hard label、soft label、hidden state 分别是什么？

hard label 是真实标签，用交叉熵让学生学习标准答案。soft label 是教师模型输出的概率分布，用 KL 散度让学生学习教师对不同类别的判断倾向。hidden state 对齐是让学生中间层表示接近教师中间层表示，通常用 MSE。

三者结合起来，学生不仅学“正确答案”，还学教师的“类间相似度”和“中间语义表达”。

Q3：蒸馏总 loss 怎么写？
loss = ALPHA * loss_hard + (1 - ALPHA) * loss_soft + BETA * loss_hid # loss_hard: 学生 logits 与真实标签做 CrossEntropy # loss_soft: 学生 logits 与教师 logits 做 KL 散度 # loss_hid : 学生 hidden state 与教师 hidden state 做 MSE # TEMPERATURE = 4.0 # ALPHA = 0.5 # BETA = 0.3 # NUM_LAYERS = 6
Q4：temperature 有什么作用？

temperature 用来平滑教师模型输出的概率分布。T 越大，概率分布越平，不会只突出最大类别。这样学生不仅知道哪个类别是正确答案，还能学到其他类别和正确类别的相似程度。

比如“膝盖疼”可能和骨科、关节科、风湿免疫科都有一定关联，soft label 可以把这种类间关系传递给学生模型。

Q5：12 层教师如何对齐 6 层学生？

一种简单做法是让学生第 i 层对齐教师第 2i+1 层，等价于从教师 12 层里隔层选取表示。比如学生第 0 层对齐教师第 1 层，学生第 1 层对齐教师第 3 层，以此类推。

这样能让学生覆盖教师从浅层到深层的语义表达，而不是只学最后一层。

Q6：知识蒸馏和量化、ONNX 有什么区别？

知识蒸馏是结构层面的压缩，把大模型的知识迁移到小模型；量化是数值精度层面的压缩，比如 FP32 变成 INT8；ONNX 更偏部署格式和推理优化。

三者可以组合使用：先蒸馏得到更小的学生模型，再导出 ONNX，最后做量化加速。

九、置信度路由追问与标准答案

Q1：你的路由逻辑具体是什么？

路由是三档：第一档 RoBERTa 学生模型先预测，如果置信度在 0.7~1，直接采纳；第二档，如果 RoBERTa 低于 0.7，就调用 RexUniNLU，RexUniNLU 置信度在 0.95~1 才采纳；第三档，如果两个模型都不满足阈值，就进入 LLM fallback。

if roberta_confidence >= 0.70: return roberta_department elif rex_confidence >= 0.95: return rex_department else: return llm_fallback_or_clarify(user_text)
Q2：为什么不用一个模型处理所有情况？

因为医疗问诊不是普通分类任务。一个模型处理所有情况会遇到三个问题：高频和低频科室分布差异大，低置信度样本强行输出容易误分诊，LLM 直接做所有分类成本高、延迟高、稳定性不够。

所以我让 RoBERTa 处理高频快速场景，RexUniNLU 复核尾部类，LLM 处理低置信度、信息不足和跨科室场景。

Q3：阈值怎么定？

不能说“凭感觉”。更合理的说法是：先在验证集上统计不同置信度区间的准确率、召回率和错误样本，再结合医疗场景的风险成本做取舍。医疗场景里误分诊成本高于多一次兜底，所以阈值要偏保守。

RoBERTa 是快速通道，阈值设置为 0.7；RexUniNLU 是高门槛复核通道，阈值设置为 0.95，只在极确定时接管。

Q4：softmax 概率能不能直接等于置信度？

严格来说不能。softmax 最大概率只是模型输出分布下的最大概率，不一定等于真实可信度，模型可能过度自信。项目里可以先把 softmax 最大概率作为工程上的置信度信号，但更完善的做法是做置信度校准，比如 temperature scaling，或者在验证集上统计各置信度区间的真实准确率。

面试时一定要说“工程置信度”，不要说“softmax 就是真实置信度”。

Q5：LLM fallback 具体做什么？

LLM fallback 不是低阈值分类器，不是让大模型强行输出一个科室。它主要处理三类情况：信息不足时追问，跨科室症状时给出谨慎建议，高危症状时输出急诊提示和免责声明。

比如用户只说“难受”，模型都低置信度，就应该追问具体症状；如果用户说“胸痛、喘不过气”，就应该先提示急诊风险，而不是只输出心内科。

十、系统设计与医疗风险控制

Q1：从用户输入“胸口疼，喘不过气”开始，完整讲一遍流程。

用户在前端输入“胸口疼，喘不过气”后，前端把文本提交给 FastAPI。后端先做基础预处理，比如去掉多余空格和特殊字符，保留关键症状。然后系统会先走高危规则判断，因为“胸口疼”和“喘不过气”都属于高风险信号，不能只当普通科室分类处理。

如果没有高危规则，系统会调用 RoBERTa 学生模型，tokenizer 把文本转成 input_ids 和 attention_mask，模型输出 logits，softmax 后得到候选科室和置信度。如果 RoBERTa 达到 0.7 以上，就采纳；否则调用 RexUniNLU；RexUniNLU 达到 0.95 以上才采纳；否则进入 LLM fallback。

但在这个例子里，因为存在急症风险，即使模型预测出心内科，也不能只推荐心内科。更安全的输出应该是：当前症状存在急症可能，建议立即前往急诊或拨打 120，同时说明可能涉及心内科、呼吸科等方向，但不能替代医生诊断。

Q2：如果医院新增一个科室，系统怎么更新？

分两种情况。如果只是知识库和规则层变化，比如新增科室介绍、医生专长、挂号说明，那么更新 RAGFlow 知识库和 MySQL 科室表，再做召回测试即可。

如果新增科室要进入模型分类，就要补充该科室训练样本，更新 class.txt 标签映射，重新构造 train/dev/test。RoBERTa 是固定分类头，新增类别通常要修改分类头并重新训练或微调；RexUniNLU 因为是 schema-driven，可以更灵活地把新增科室加入 schema，但要稳定上线仍然需要样本微调和评估。

Q3：模型预测错了怎么追溯？

需要记录请求日志和模型输出链路，包括用户输入、预处理文本、RoBERTa top-k 结果和置信度、RexUniNLU 结果和置信度、是否触发 LLM fallback、最终输出科室和建议。

这样出现误分诊时，可以判断是数据问题、模型问题、阈值问题、规则缺失还是知识库召回错误。后续可以把低置信度和人工纠错样本回流到数据集，做持续优化。

Q4：医疗场景如何避免过度承诺？

第一，所有输出都定位为导诊辅助，不说“诊断为”。第二，输出建议时使用“可能涉及”“建议优先考虑”“请结合医生诊断”这类谨慎表达。第三，高危症状优先急诊提示。第四，所有医疗建议追加免责声明。第五，低置信度场景不强行分类，而是追问或转人工。
