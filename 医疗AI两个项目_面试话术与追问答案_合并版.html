项目一追问答案：挂号助手 / RAG / Dify

1. 你说用了 RAG，完整流程是什么？

用户输入后，系统先判断意图，比如是科室推荐、药品说明、费用查询还是急救知识。然后根据用户问题去对应知识库里检索相关片段，比如科室介绍、医生专长、疾病与科室关系、药品用法禁忌等。

检索到资料后，系统会把用户问题和召回内容一起放入 prompt，让大模型基于资料回答，而不是自由发挥。最后输出时会加免责声明；如果检测到胸痛、呼吸困难、意识模糊这类高危信号，则优先提示急诊或 120。

2. 为什么不用大模型直接回答？

医疗场景对准确性和可控性要求更高。直接让大模型回答，容易出现幻觉，比如编造医生信息、错误推荐科室、给出不合规用药建议。

RAG 的作用是先检索医院或业务方已有资料，再让大模型基于资料生成回答。这样回答范围更可控，也方便后续追溯问题来源。

3. 知识库里放了什么？怎么组织？

主要包括科室介绍、医生专长、常见疾病与科室关系、药品说明书、急救知识、挂号说明等资料。组织时会尽量按业务类型分库，比如科室推荐知识库、药品说明书库、急救知识库，避免不同类型资料互相干扰。

例如用户问“牙痛挂什么科”，系统优先召回口腔科、牙体牙髓、口腔颌面相关资料；用户问“布洛芬怎么吃”，则进入药品说明知识库，重点输出用法、禁忌、最大剂量和注意事项。

4. 用户只说“肚子疼”，你怎么处理？

这种属于信息不足，不应该直接给确定科室。系统应该追问疼痛部位、持续时间、是否伴随发热、呕吐、腹泻、便血、怀孕可能等信息。

如果用户描述里出现持续剧烈腹痛、意识异常、呕血便血等危险信号，就优先提示急诊，而不是继续普通挂号推荐。

5. 面试官问“准确率提升多少”，没有数据怎么办？

不要硬编数字。可以说：这个项目更多是应用流程验证，不是严格模型评测。我主要从问答相关性和业务可用性角度验证，比如典型挂号问题能否召回正确资料、回答是否符合科室知识、是否触发高危规则。

如果要正式量化，我会建立标准测试集，用检索命中率、回答相关性、人工评估通过率、高危规则召回率来衡量，而不是简单说“准确率提升多少”。

6. Coze 和 Dify 你具体做了什么？

我主要做的是对话流程配置和节点编排，包括 LLM 意图识别节点、条件分支、知识库检索节点、结构化查询节点和回复节点。还会根据业务场景配置不同 prompt，让系统在不同意图下走不同路径。

例如药品类问题走药品知识库，科室推荐问题走科室知识库，价格类问题走 MySQL 服务价格表，高危症状直接触发规则提示。

项目二：AI 智能问诊分诊系统

这个项目的关键词是：数据工程、长尾分类、RoBERTa-wwm-ext、RexUniNLU、知识蒸馏、置信度路由、LLM fallback。这是你真正体现“AI 应用开发 + 模型链路理解”的核心项目。

稳妥版项目介绍

这个项目面向医院导诊和互联网医疗场景，目标是根据用户症状文本预测推荐科室，并生成专家式健康建议。项目难点主要是医疗文本表达不标准、科室类别长尾严重，以及低置信度误分诊风险高。

我的工作主要集中在数据工程和模型链路设计。数据上，我处理约 25 万条问诊文本，做清洗、去重、标签冲突处理、科室名防泄漏，并针对小样本科室做补齐。模型上，我用 RoBERTa-wwm-ext 训练 24 类科室分类器，并设计 6 层学生模型做知识蒸馏；同时把 20 个低资源科室改写成 RexUniNLU schema-driven 格式做二次判断。最后通过置信度路由，把 RoBERTa、RexUniNLU 和 LLM 串起来，兼顾速度、准确率和医疗安全边界。

项目二追问答案：数据、模型、蒸馏、路由

1. 为什么选 RoBERTa-wwm-ext？

因为它适合中文短文本语义理解。wwm 是 whole word masking，也就是整词 Mask。中文医疗症状里很多词是多字词，比如“呼吸困难”“胸口疼”“关节疼痛”，整词 Mask 比单字 Mask 更能帮助模型学习词级语义。

项目里 RoBERTa 承担头部和腰部科室的快速分类，结构是 RoBERTa 主干加线性分类头，优点是稳定、推理快、工程实现简单。

2. 为什么强调 Macro F1，而不是 Accuracy？

因为医疗科室分类是典型长尾任务。如果只看 Accuracy，模型把头部科室分好，整体指标就可能很好看，但尾部科室可能完全错。Macro F1 是对每个类别分别计算 F1 后再平均，更能反映尾部类别表现。

所以我会用 Accuracy 看整体命中率，用 Macro F1 判断模型是否照顾到低频科室。

3. 标签冲突怎么处理？

同一条症状文本如果对应多个不同科室，就属于标签冲突。这类样本会给模型互相矛盾的监督信号。处理方式是先用文本归一化后分组，检查同文本不同标签的情况；如果有人工条件，优先复核；如果没有，就先删除这类样本，保证训练信号干净。

4. 为什么要做科室名防泄漏？

因为真实用户通常不会说“我要挂骨科”，而是说“膝盖疼、走路疼”。如果训练文本里直接出现科室名，模型会记住捷径，导致测试指标虚高，但真实场景泛化差。

所以要清理“骨科、眼科、皮肤科”等科室名直出内容，让模型更多依据症状、部位、伴随表现来判断。

5. RexUniNLU 的 schema-driven 是什么意思？

schema-driven 的意思是：任务由 schema 定义，模型根据输入文本和 schema 判断输出类型。我们把传统分类任务改写成识别 [CLASSIFY] 这个 span 属于哪个科室类型。

每条样本里会有 text、info_list 和 schema。schema 里列出 20 个候选科室，info_list 里标注正确科室。这样 RexUniNLU 就能把分类任务当成统一信息抽取任务来微调。

6. RexUniNLU 指标 0.9971 会不会太高？怎么解释？

会被质疑，所以不能把它说成全场景效果。正确解释是：这是针对筛选出的 20 个低资源科室子任务，测试集 2400 条，每类约 120 条，部分类别语义锚点比较明确，所以指标很高。

它不等价于全 58 类真实场景效果。真实系统里仍然保留 RoBERTa + RexUniNLU + LLM 的级联路由，就是为了应对全量分布、模糊症状和跨科室表达。

7. 知识蒸馏三种 loss 怎么讲？

hard label loss 是学生模型和真实标签之间的交叉熵，让学生学习标准答案。

soft label loss 是学生 logits 和教师 logits 之间的 KL 散度，通过 temperature 让教师输出分布更平滑，学生可以学到类间相似度。

hidden state loss 是学生隐层表示和教师隐层表示之间的 MSE。项目里学生第 i 层对齐教师第 2i+1 层的 [CLS] 表示，让学生不仅学输出，也学中间语义表达。

loss = ALPHA * loss_hard + (1 - ALPHA) * loss_soft + BETA * loss_hid
8. 你的置信度路由逻辑是什么？

第一层先调用 RoBERTa 学生模型，如果它的 softmax 最大概率在 0.7~1，就采纳对应科室；如果低于 0.7，不直接输出，进入 RexUniNLU。

第二层 RexUniNLU 独立计算自己的置信度，只有 0.95~1 才采纳。这个阈值更高，是因为医疗误分诊成本高，尾部类复核必须更谨慎。

第三层，如果两个模型都不满足阈值，就进入 LLM fallback，让 LLM 做进一步询问、解释和风险提示，而不是强行输出一个低可信度科室。

9. softmax 最大概率能直接当置信度吗？

严格来说不能完全等同。softmax 最大概率只是模型输出分布下的工程置信度信号，模型可能过度自信。

所以面试时要说：项目里先用 softmax 最大概率作为路由信号，但更完善的做法是用验证集做置信度校准，例如统计不同置信度区间的真实准确率，或者做 temperature scaling。

10. “胸口疼，喘不过气”怎么走完整流程？

用户输入后，系统先做基础预处理，然后优先走高危规则判断。因为“胸口疼”和“喘不过气”属于高风险信号，不能直接只推荐心内科。

正常分类链路会先进入 RoBERTa，如果置信度达标则得到候选科室；不达标则进入 RexUniNLU；再不达标进入 LLM fallback。但这个例子中，高危规则优先级应该高于普通分诊输出。最终回答应提示：当前症状存在急症可能，建议立即前往急诊或拨打 120，同时说明可能涉及心内科、呼吸科等方向，但不能替代医生诊断。

最终背诵版：两个项目合并讲述

这一段适合你在面试中讲完整项目经历。尽量讲自然，不要一字不差背稿。

我在实习期间主要参与了两个医疗 AI 项目。第一个是智能医疗挂号 / 预问诊助手，偏大模型应用落地；第二个是 AI 智能问诊分诊系统，偏模型训练和工程化路由。

第一个项目里，我主要参与 Coze、Dify 对话流程搭建和 RAG 知识库接入。我们把科室介绍、医生专长、疾病与科室关系、药品说明和急救知识整理成知识库，用户提问后先识别意图，再检索相关资料，最后由大模型基于资料生成回复。这样做不是为了让大模型直接诊断，而是让它作为导诊辅助工具，提升回答相关性和可控性，并通过高危规则和免责声明控制医疗风险。

第二个项目是智能问诊分诊系统，目标是根据用户症状文本预测推荐科室。这个项目里我重点做了数据工程和模型链路设计。数据上处理约 25 万条问诊文本，包括去空文本、去重、标签冲突处理、科室名防泄漏和长尾科室补齐。模型上用 RoBERTa-wwm-ext 做 24 类科室分类器，测试集 Macro F1 是 0.8497；为了提升部署效率，又做了 6 层学生模型蒸馏，用 hard label、soft label 和 hidden state 对齐保留教师知识。

对于低资源科室，我引入 RexUniNLU 做二次判断，把 20 类任务改写成 schema-driven JSONL 格式，通过 [CLASSIFY]、info_list 和 schema 表达分类任务，测试集 Macro F1 是 0.9971。不过我会把这个结果限定为 20 类低资源子任务，不会等同于全 58 类真实场景效果。

最后系统采用三级置信度路由：RoBERTa 置信度达到 0.7 以上才采纳；否则进入 RexUniNLU，RexUniNLU 达到 0.95 以上才采纳；两个模型都不满足时，进入 LLM fallback 做进一步询问和风险提示。这个设计的核心是让每个模型只在自己足够确定的场景输出，从而兼顾高频场景速度、尾部科室覆盖和医疗安全边界。
